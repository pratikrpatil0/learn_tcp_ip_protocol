\chapter{Introduction}
A major issue with computer networks is that they are not always reliable. At
first, sending data from one computer to another may seem easy. In practice,
real communication has to deal with packet loss, duplication, corruption,
delay variation, and delivery that is not in order. TCP and other transport
protocols were made to fix these problems. However, the behavior of TCP can
feel abstract when studied only through theory.

This project adddresses that learning gap by gradually adding transport
protocol features to UDP sockets in C. The project doesn't use the operating
system's TCP stack directly. Instead, it builds many TCP-like ideas into the
application code. This method shows every design choice. The logic for sending
and receiving, managing sequences, handling timeouts, and responding to
congestion are all clearly defined and followed.

There are twenty-five sender-receiver versions in the full work, from level 0
to level 24. The first levels use bit transmission and byte reconstruction.
The middle levels add error detection and reliability that stops and waits. The
advanced levels have ideas for buffering that are like TCP Reno behavior, such
as handshake, checksum, sliding window, selective-repeat style buffering,
adaptive timeout estimation, timestamp, and SACK-like ideas.

\section{Problem Statement}
The project aims to answer one practical question:

\begin{displayquote}\raggedright\doublespacing
How can we incrementally construct a reliable and adaptive transport protocol over unreliable UDP datagrams while keeping the implementation understandable for learning and experimentation, while also using AI tools responsibly for idea support (not code copying) and validating every design step through standard networking textbooks?
\end{displayquote}

\vspace{\baselineskip}

UDP only guarantees best-effort delivery of datagrams. It doesn't make sure
that packets arrive, that they are in the right order, that duplicates are
removed, that congestion is controlled, or that the connection state is
correct. So, the application layer has to use these features if they are
needed. This project shows how to do that step by step. The learning method in
this work is book-first and concept-first: AI tools were only used to talk
about other options, help with debugging, and come up with pseudocode ideas.
The final logic, code writing, and verification were all done by hand.

The conceptual path follows a top-down networking approach in the spirit of
widely used teaching references, especially Kurose and Ross for protocol
layering and transport intuition, Stevens for implementation-level thinking
about TCP/IP behavior, and Tanenbaum and Wetherall for broader networking
foundations \cite{kurose_ross,stevens_tcpip,tanenbaum_wetherall,rfc793,rfc6298,rfc5681,rfc2018,rfc7323,rfc768,beej_guide}.

\section{Project Objectives}
The major objectives of the project are listed below.

\begin{enumerate}
	\item Build a complete progression from raw transmission to transport
	control.
	\item Keep each version self-contained, so learners can compare behavior
	across levels.
	\item Follow a responsible learning process: study textbook concepts first,
	then use AI tools for idea exploration, and finally implement and test
	independently without copying generated code.
	\item Implement reliability features step by step: parity, ACK/NAK,
	timeout retries, sequence control, and retransmission logic.
	\item Move from one-byte packets to chunk-based payload transfer to improve
	throughput.
	\item Introduce adaptive timing and congestion-aware sending to mimic
	practical transport design.
	\item Document implementation, assumptions, limitations, and future
	improvements in a technical report.
\end{enumerate}

\section{System Context and Setup}
All implementations use IPv4 UDP sockets (\texttt{AF\_INET} with
\texttt{SOCK\_DGRAM}) on port 9090. The sender transmits to localhost
(\texttt{127.0.0.1}), and the receiver binds to \texttt{INADDR\_ANY}. Typical
message size in this project is bounded to around 99 characters.

The same high-level execution pattern is followed in all versions:

\begin{enumerate}
	\item Start the receiver first.
	\item Then start the sender.
	\item Type the message in the sender terminal.
	\item Watch the transfer, ACKs, retries, and finish.
\end{enumerate}

\section{Version Progression Strategy}
The project does not attempt to implement all features at once. Instead, it
follows a layered progression model:

\begin{enumerate}
	\item \textbf{Foundation phase (v0--v2):} bit and byte transfer.
	\item \textbf{Reliability phase (v3--v7):} parity, sequence tracking,
	stop-and-wait, timeout retries, packet structure.
	\item \textbf{Transport phase (v8--v14):} handshake, checksum,
	connection close, sliding window, chunking.
	\item \textbf{Adaptive and Reno-like phase (v15--v24):} adaptive RTO,
	timestamps, SACK-style hints, congestion control, fast retransmit, and
	recovery.
\end{enumerate}

\section{Educational Value}
From an academic viewpoint, this project offers a practical link between computer networks, systems programming, and algorithm design. Key ideas from textbooks become tangible:

\begin{enumerate}
	\item \textbf{Data integrity:} parity and checksum validation.
	\item \textbf{Reliability:} acknowledgment and retransmission.
	\item \textbf{Ordering:} sequence numbers and expected sequence tracking.
	\item \textbf{Pipelining:} multiple in-flight packets through windows.
	\item \textbf{Flow under uncertainty:} adaptive timeout and congestion
	response.
\end{enumerate}

\vspace{\baselineskip}

Since each feature is implemented manually, learners can see trade-offs directly. For instance, stop-and-wait is simple to understand, but it is inefficient at higher latency. Sliding windows boost throughput but increase state complexity. Selective-repeat buffering enhances reliability under reordering but requires careful tracking.

The interactive visualizer further adds educational value by displaying the sender code, receiver code, finite state machine, and UML timeline together. This approach makes it easier for beginners to grasp protocol state transitions as they learn transport concepts for the first time.

\section{Scope and Boundaries}
The project is designed for learning and controlled experiments. Important
boundaries are listed below.

\begin{enumerate}
	\item Communication is mainly localhost based.
	\item Address and port are hardcoded in most versions.
	\item Security features like encryption and authentication are not
	included.
	\item Receiver-advertised window and full SACK format are not fully
	implemented.
	\item Most versions handle one sender-receiver flow at a time.
\end{enumerate}

These boundaries are acceptable for a graduation project focused on protocol
mechanics. They also provide clear directions for future extension.

\section{Research Questions}
The implementation and analysis are organized around practical questions:

\begin{enumerate}
	\item How much reliability can be achieved using only user-space logic over
	UDP?
	\item At which stages does throughput improve significantly, and why?
	\item How does adaptive timeout outperform fixed timeout under variable
	delays?
	\item How does congestion window control affect retransmission behavior and
	completion time?
	\item How can students use AI tools responsibly for protocol learning
	(idea-level guidance and debugging support) while preserving original
	implementation ownership and textbook-grounded correctness?
\end{enumerate}
